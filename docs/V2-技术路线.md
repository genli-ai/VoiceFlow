# VoiceFlow V2 技术路线：升级本地识别引擎

> 目标：用 2026 年开源 SOTA 的 Qwen3-ASR 替换/并存 whisper，根治中文及中英混说（尤其专有名词、英文缩写）的识别短板。
> 状态：调研完成，待开工。调研日期：2026-06-10

## 一、为什么升级

当前 V1 使用 whisper large-v3-turbo（2024.10），仍是 whisper 家族最强，但：

- 中文里夹英文缩写（UAE、API 等）和同音人名是其弱项，实测多次出错
- 阿里 2026.1 发布的 **Qwen3-ASR** 在几乎所有公开基准上超过 whisper-large-v3，支持 52 种语言 + 22 种中文方言，中文/中英混说是其最强项

## 二、候选方案

### 方案 A（主推荐）：Qwen3-ASR + MLX Swift

集成 [ontypehq/mlx-swift-asr](https://github.com/ontypehq/mlx-swift-asr)（MIT 协议）——OnType 公司（macOS 语音输入工具商）开源的 Swift 原生引擎，与我们场景完全同构：

- 支持 Qwen3-ASR 0.6B / 1.7B，**0.6B-6bit 仅约 400MB**（比现在的 whisper 模型还小 150MB）
- 实测 M1 Pro 上 RTF 0.15-0.27（**3.7-6.3 倍实时**），比我们现在的 whisper+束搜索更快
- Actor 线程安全 API、内置 16kHz 重采样、Metal shader 预热（消除首次延迟）
- 梅尔频谱与 Whisper 前端一致 → 我们的 AudioRecorder 无需改动

**硬约束（必须知晓）：**

| 约束 | 影响 |
| --- | --- |
| MLX 的 Metal 着色器**无法用 `swift build` 编译**，必须 `xcodebuild` | 构建机需要完整 Xcode（约 10GB，免费）；**用户侧分发模式要从"源码编译"改为"预编译二进制"**（GitHub Releases 直接下 .app） |
| 要求 macOS 15+、Swift 6.2+、Apple Silicon | Intel 与老系统用户继续用 whisper 引擎（双引擎并存） |
| 仓库很年轻（3 commits） | 按 MIT 协议 fork/vendor 一份锁定版本，问题自己可修 |
| 词汇表偏置（context biasing）API 未确认 | Qwen3-ASR 模型本身支持上下文热词，但此 Swift 库未暴露；需验证，必要时自己加（工作量可控，值得做——这是杀手锏） |

### 方案 B（备选）：SenseVoice + sherpa-onnx

若不接受"必须装 Xcode/二进制分发"，可用 sherpa-onnx（提供预编译库，CLT 友好）跑阿里 SenseVoice-Small：中文很强、速度极快，但准确率属于上一代（2024），低于 Qwen3-ASR。**仅当方案 A 受阻时启用。**

## 三、实施计划（方案 A）

1. **引擎抽象层**（半天）：`SpeechEngine` 协议；现有 WhisperService 改为 `WhisperEngine`；设置页加引擎选择（自动：Apple Silicon + macOS 15 → Qwen，否则 whisper）
2. **QwenEngine 集成**（1 天）：接入 MLXASR、模型加载/预热/转写、与现有词汇表 prompt 机制对齐（验证 context biasing）
3. **模型下载器扩展**（半天）：Qwen 模型是多文件目录（safetensors+config），扩展 ModelDownloader 支持清单式多文件下载，hf-mirror 优先
4. **构建与分发改造**（1 天）：installer 改 xcodebuild 路径；产出 arm64 .app 上传 GitHub Releases；保留 V1 源码编译路径给 whisper-only 用户；（可选）Apple Developer ID $99/年 → 公证，用户免"右键打开"
5. **对比测试与发布**（半天）：同一批真实语音（中英混说、UAE/人名、轻声）双引擎对比，确认胜出再设为默认

预估总量：3-4 个工作日的对话迭代。

## 四、风险与回退

- 任何阶段受阻 → V1 whisper 路径完整保留，用户无感
- mlx-swift-asr 缺陷 → 已 vendor，可自修；最坏回退方案 B
- 性能不及预期 → 双引擎设置项让用户自选
